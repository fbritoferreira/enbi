# 41. Astro admin functional core: login, collections, entry CRUD, and API-key management

- Status: Accepted
- Date: 2026-06-20

## Context

The Astro admin app (`apps/admin`) was scaffolded in ADR 0002 and ADR 0029. The scaffolding
established the framework choice and the mechanism by which the CLI drives Astro programmatically,
but the admin's pages contained only placeholder content — no working login flow, no collection
browser, no entry editor, and no API-key UI.

Several cross-cutting concerns needed to be resolved before building those pages:

**Collection metadata over HTTP.** The admin must know which collections exist and what fields they
have in order to render a collection list and entry forms. Reading `enbi.config.ts` directly from
the admin would require the admin to import user code at build time, coupling the admin's static
build to the project's runtime config. Instead, a dedicated server endpoint
`GET /api/admin_collections` returns collection metadata as JSON; the admin fetches it at runtime
and never imports user config.

**Cross-origin cookies.** In a production deployment the admin and the server are separate origins
(e.g. `admin.example.com` vs `api.example.com`). In development and testing they share the same
host (`localhost`) — different ports, but the same _site_ — which means the session cookie set by
the server is classified as **same-site** under the `SameSite=Lax` default and is therefore sent
on cross-origin fetch requests from the admin. To support this correctly and securely the server
must also:

- Add **CORS** headers (specific `Access-Control-Allow-Origin` matching the configured admin
  origin, plus `Access-Control-Allow-Credentials: true`).
- Register the admin origin in better-auth's **`trustedOrigins`** list, which is better-auth's
  CSRF allowlist for cross-origin requests.

**Static routing.** Astro's SSG mode generates HTML files at build time. Dynamic route segments
(`[collection].astro`) require `getStaticPaths()` to enumerate all possible values; at build time
the collection list is not known, so `getStaticPaths` would return an empty array and every
collection URL would 404 in production. The admin pages therefore use **query strings** rather than
dynamic path segments (`/entries?c=<collection>`, `/edit?c=<collection>&id=<id>`). Page files are
non-dynamic (`entries.astro`, `edit.astro`) and are always generated by SSG.

## Decision

The Astro admin gains a functional core consisting of:

- **Login page** — authenticates via the server's better-auth session endpoint; the session cookie
  is stored by the browser and sent on subsequent API calls.
- **Collections index** — fetches `GET /api/admin_collections` and renders a list of available
  collections.
- **Entry list** — fetches `GET /api/<collection>` (with `?limit`/`?offset` pagination from
  ADR 0038) and renders a table of entries for the selected collection.
- **Entry create/edit form** — renders fields from the `admin_collections` metadata; submits to
  `POST /api/<collection>` or `PUT /api/<collection>/:id`.
- **API-key management** — lists, creates, and revokes API keys via the HTTP key-management
  endpoints from ADR 0034.

The server adds:

- `GET /api/admin_collections` — returns an array of `{ name, fields }` objects; accessible only
  to authenticated admin-role sessions.
- CORS middleware scoped to the configured admin origin with `credentials: true`.
- `trustedOrigins` passed through `buildAuthOptions`/`createAuth` so better-auth accepts
  cross-origin session requests from the admin origin. `EnbiAuthOptions` gains a
  `trustedOrigins?: string[]` field for this purpose.

A production deployment on **different registered domains** (not just different ports on the same
host) would require `SameSite=None; Secure` cookies, which in turn requires HTTPS on both origins.
This is noted as a deployment concern but is out of scope for the current implementation, which
targets `localhost`-based dev/test environments.

Admin pages are static (`entries.astro`, `edit.astro`) and use query strings for collection and
entry identity. Dynamic Astro routes are not used.

## Consequences

- **Good:** the admin is fully operational for the primary content-management workflows without
  requiring a separate BFF or build-time config injection.
- **Good:** `GET /api/admin_collections` keeps the admin decoupled from user config at build time;
  the admin bundle is a pure static artifact.
- **Good:** the CORS + `trustedOrigins` combination is correct and explicit; neither is silently
  assumed.
- **Good:** SSG with query-string routing avoids the empty-`getStaticPaths` 404 problem.
- **Operational concern:** production deployments on different registered domains need
  `SameSite=None; Secure` cookies and HTTPS. This is a known limitation that must be addressed
  before cross-domain production use.
- **API surface:** `EnbiAuthOptions.trustedOrigins` and the `trustedOrigins` parameter to
  `buildAuthOptions`/`createAuth` are new public API; this is a minor addition to `@enbi/auth`.

## Alternatives considered

- **Import `enbi.config.ts` in the admin at build time:** would allow SSG pages to know the
  collection list but couples the static build to user runtime code. Rejected — the HTTP metadata
  endpoint is cleaner and keeps the admin bundle generic.
- **Dynamic Astro routes with `getStaticPaths`:** natural for Astro but impossible when the data
  is only available at runtime. Rejected in favour of query-string routing on static pages.
- **Proxy the admin through the server (same origin):** eliminates the CORS/cookie complexity but
  requires the server to serve Astro's static output, coupling two independently deployable
  services. Rejected — separate origins with explicit CORS is the right long-term model.
- **`SameSite=None; Secure` cookies from the start:** correct for cross-domain production but
  requires HTTPS in development, adding setup friction. Deferred until a production cross-domain
  deployment is configured.

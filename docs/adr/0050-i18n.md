# 50. Field-level i18n via `_translations`

- Status: Accepted
- Date: 2026-06-20

## Context

CMS content needs to be published in multiple languages. The standard
approaches are:

1. **Duplicate rows** — a separate row per locale in the same collection.
2. **JSON column** — a single JSONB column holding all translations.
3. **Side-table** — a `_translations` table keyed by `(collection, entryId, locale, field)`.

Row duplication couples content structure to locales, making queries
cumbersome and draft/publish logic per-locale complex. A JSON column is
opaque to SQL queries and migrations. The side-table approach keeps the
primary collection table unchanged, allows per-field opt-in, and is fully
additive.

## Decision

### `_translations` table — `@enbi/db`

A new internal per-dialect table mirroring the `_media`, `_revisions`, and
`_api_keys` tables:

```sql
_translations (
  id         text PRIMARY KEY,
  collection text NOT NULL,
  entry_id   text NOT NULL,
  locale     text NOT NULL,
  field      text NOT NULL,
  value      text
)
```

The tuple `(collection, entry_id, locale, field)` is the logical unique key.
Upsert is implemented as delete-then-insert for cross-dialect compatibility
(SQLite `ON CONFLICT DO UPDATE` syntax differs from PostgreSQL and MySQL).
`TranslationsTable` and `translationsFor(dialect)` follow the identical
pattern of `mediaFor`, `revisionsFor`, etc.

### Config — `@enbi/db`

`EnbiConfig` gains an optional field:

```typescript
i18n?: { locales: string[]; defaultLocale?: string }
```

`CollectionOptions` gains:

```typescript
localized?: string[]  // names of translatable fields; defaults to []
```

`Collection` (resolved type) carries:

```typescript
localized: string[]   // defaults to []
```

### Read overlay — `apps/server`

`GET /api/:col` and `GET /api/:col/:id` support `?locale=<code>`. When
`locale` is present:

- If the locale is not in `config.i18n.locales`, the server returns **400**.
- For each returned row, each field listed in `col.localized` is looked up
  in `_translations`. If a row exists, its value replaces the base field
  value. If not, the base value is preserved (default-locale fallback).
- `locale` is added to the reserved query-parameter set so it is never
  interpreted as a filter.

### Translations endpoints — `apps/server`

```
GET  /api/:col/:id/translations/:locale   (auth: read)
PUT  /api/:col/:id/translations/:locale   (auth: update)
```

`PUT` body: `{ [field]: value, ... }`. Each key must be in `col.localized`
(else **422**); locale must be configured (else **400**). Returns the stored
translations for the tuple.

### Admin UI — `@enbi/admin`

`edit.astro` gains a locale `<select>` when the collection has `localized`
fields and i18n locales are configured. Selecting a non-default locale fetches
the existing translations and populates the localized fields. Save writes via
the translations `PUT` instead of the normal row `PUT`. Default-locale edits
continue to write the base row.

The `/api/admin_collections` metadata response now includes:

```json
{
  "collections": [...],
  "locales": ["en", "fr"],
  "defaultLocale": "en"
}
```

Each collection entry now includes `"localized": ["title", "body"]`.

## Consequences

- **Good:** additive — collections without `localized` are unaffected.
- **Good:** per-field opt-in; non-string or system fields stay in the base row.
- **Good:** admin locale switcher is minimal and functional without extra dependencies.
- **Bad (N+1):** the read overlay performs one DB query per row to fetch
  translations. This is acceptable for typical CMS page sizes (tens to low
  hundreds of rows), but a batch-by-entryId query could reduce it to one
  query per list response. Deferred; noted here for future improvement.
- **Bad:** no unique constraint on `(collection, entry_id, locale, field)` at
  the DB level — enforced by the delete-then-insert upsert pattern instead.

## Non-goals

- **Locale routing / Accept-Language negotiation** — locale must be explicit
  in the query parameter; browser headers are not consulted.
- **Per-locale publishing** — draft/publish state remains on the base row
  and applies to all locales equally.
- **RTL / formatting / date localisation** — purely content-storage layer.
- **Translating non-string fields** — `value` is stored and returned as text;
  numeric or boolean fields are not supported in `localized`.
